# 第一次 Git 提交流程

这份说明专门给你这种“以前主要在 GitHub 网页上改文件”的使用方式准备。

目标是把现在这个本地项目，变成以后可以稳定维护的 Git 项目。

## 1. 先判断你属于哪一种情况

### 情况 A：GitHub 上已经有这个项目仓库，而且 Cloudflare 正在用它自动部署

这是最常见的情况。

这种情况下，不要新建第二个同名项目，也不要乱覆盖历史。

你只需要把当前这个本地目录，连接到你现在正在用的 GitHub 仓库。

### 情况 B：GitHub 上还没有正式仓库，或者你准备重新建一个新的

这种情况下，可以新建一个空仓库，再把本地项目第一次推上去。

## 2. 最推荐的做法

最推荐你以后用：

- GitHub Desktop 管理提交和推送

原因很简单：

- 不容易输错命令
- 能直接看到改了哪些文件
- 回滚更直观
- 对新手最友好

如果你不想装 GitHub Desktop，也可以继续用 GitHub 网页上传，但以后最好只上传我告诉你的变更文件。

## 3. 第一次提交的标准命令

如果你愿意用命令行，第一次提交一般是这四步：

```bash
git add .
git commit -m "chore: initialize project management workflow"
git remote add origin 你的GitHub仓库地址
git push -u origin main
```

## 4. 如果 GitHub 上已经有旧仓库

如果你的 GitHub 仓库里已经有这套代码，不要先急着执行 `git push`。

更稳的方式是：

1. 先把仓库地址发给我
2. 我帮你判断应该直接推、先拉取，还是先做一次备份

这是为了避免把线上仓库历史弄乱。

## 5. 以后每次修改后的固定提交流程

以后不需要每改一个地方就上传一次。

更推荐这样做：

1. 你把需求一次性告诉我
2. 我在本地连续修改
3. 我先帮你做本地检查和模拟测试
4. 我确认这轮改动完成后，再告诉你该同步哪些文件
5. 你最后一次性提交并推送到 GitHub

## 6. 你以后最常见的三种动作

### 只看改了什么

```bash
git status
```

### 提交这次改动

```bash
git add .
git commit -m "feat: 本次功能说明"
```

### 推送到 GitHub

```bash
git push
```

## 7. 最重要的一条

以后默认流程不是：

“改一点，传一点，再去网页测，再回来改一点”

而是：

“你集中提需求 -> 我集中修改和测试 -> 你最后统一同步到 GitHub -> Cloudflare 自动部署 -> 你做最终验收”
