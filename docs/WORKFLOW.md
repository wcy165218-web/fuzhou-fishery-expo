# 以后和 Codex 的协作工作流

## 1. 你负责什么

你只需要做三件事：

1. 提需求或报问题。
2. 在 Cloudflare 或网页上做最后验证。
3. 如果这次改动涉及 D1 / R2 控制台，我按步骤写给你后，你照着点。

你不需要自己拼代码，也不需要自己判断改哪几个文件。

## 2. 我负责什么

我负责：

- 阅读现有代码
- 直接修改项目文件
- 说明改动影响
- 告诉你这次改了哪些文件
- 告诉你下一步要测试什么
- 如果有 D1 / R2 改动，给你一份可直接照做的操作清单

## 3. 每次改动后的标准流程

1. 你告诉我需求、报错、截图或预期效果。
2. 我直接改本地项目代码。
3. 我告诉你：
   - 改了什么
   - 改了哪些文件
   - 是否涉及 D1 / R2
   - 你要验证哪些地方
4. 你只同步这次改动过的文件到 GitHub。
5. 等 Cloudflare 自动部署完成后再测试线上页面。

重点：不是每次都要把所有文件重新覆盖到 GitHub。

只要上传这次真正改过的文件就可以。

更推荐的节奏是：

1. 你把这一轮需求一次性告诉我
2. 我在本地连续改完
3. 我先帮你做模拟测试和基础检查
4. 这一轮都处理完以后，你再统一同步到 GitHub 一次

这比“每改一点就立刻传 GitHub”更省心，也更不容易把中间状态传上去。

## 4. 测试分成两层

### A. 结构检查

这是最基础的检查，目的是确认代码没有明显语法错误。

常用命令：

```bash
npm run check
```

### B. 联调测试

如果要测试登录、D1 查询、R2 上传、订单流程这种真实业务，就需要 Cloudflare 绑定齐全。

等 `wrangler.toml` 配好以后，可以用：

```bash
npm run dev
```

本地打开 Wrangler 给出的地址测试。

如果你要连真实的 Cloudflare 资源联调，可以用：

```bash
npm run dev:remote
```

注意：

- `npm run dev` 更适合本地预览代码改动
- `npm run dev:remote` 会更接近真实环境，但要更谨慎，避免误操作真实数据

## 5. D1 / R2 改动以后怎么做

### D1

以后凡是数据库改动，我都会：

1. 在 `migrations/` 里留 SQL 文件。
2. 在回复里告诉你执行顺序。
3. 标明是“新增字段”“新增表”“补数据”还是“修数据”。

### R2

以后凡是对象存储改动，我会告诉你：

1. 要不要新建 bucket
2. bucket 名称是什么
3. Worker 绑定名是什么
4. 是否需要改 CORS 或公开访问策略

## 6. 最推荐的发布方式

最推荐的是：

1. 这个文件夹本地保留完整项目。
2. 用 Git 记录改动。
3. 每次只提交变更文件。
4. 推送到 GitHub。
5. 等 Cloudflare 自动部署。

如果你暂时还是习惯 GitHub 网页上传，也可以只覆盖本次变更过的文件，不需要每次重传整个项目。

第一次 Git 提交流程见 [docs/GIT-FIRST-COMMIT.md](/Users/wangchuanyi/Downloads/fuzhou-fishery-expo-main/docs/GIT-FIRST-COMMIT.md)。

## 7. 你提需求时最好这样发

最好带上这些信息：

- 你想改哪个页面或哪个功能
- 现在看到的现象
- 你希望它变成什么样
- 如果报错，附文字或截图
- 如果涉及数据，说明是 D1 还是 R2

例如：

- “订单列表加一个按业务员筛选”
- “上传合同后提示文件不存在”
- “项目配置页新增一个封面图上传，文件放 R2”
